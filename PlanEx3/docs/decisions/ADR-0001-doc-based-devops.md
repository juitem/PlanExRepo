## Context (맥락)

이 프로젝트는 여러 AI 에이전트가 순차적/병렬적으로 개발을 수행한다. 에이전트는 언제든 중단될 수 있고, 다른 에이전트가 이를 이어받아야 한다. 단, 에이전트는 이전 대화 기록이나 메모리를 공유하지 않는다.

일반적인 소프트웨어 개발 방식(즉흥적 코딩, 암묵적 컨텍스트 의존)은 이 방식에 적합하지 않다. 에이전트가 교체되거나 재시작되면 작업 컨텍스트가 소실된다.

**해결해야 할 문제**:
- 에이전트 교체 시 컨텍스트 소실
- 병렬 작업 시 충돌 및 중복
- 의사결정 근거의 소실
- 진행 상태의 불투명성

---

## Decision (결정)

**`PlanEx3/docs/` 아래에 구조화된 markdown 문서 패키지를 두고, 모든 작업 컨텍스트, 상태, 의사결정을 이 문서들로 관리한다.**

구체적으로:
1. 모든 에이전트는 작업 시작 전 `00_INDEX.md` → `CURRENT_STATUS.md` → 역할 문서 순으로 읽는다
2. 모든 아키텍처 결정은 ADR(Architecture Decision Record)로 기록한다
3. 지식은 `stable / working / episodic / decisions / interface` 5개 레이어로 분류한다
4. 모든 Job은 입력/출력/완료조건을 문서화한 실행 계약으로 정의한다
5. 모든 상태 변경은 `CURRENT_STATUS.md` 와 `CHECKPOINT_LOG.md` 에 즉시 반영한다

---

## Rationale (근거)

1. **재진입 가능성**: 문서 기반이면 에이전트가 대화 기록 없이도 작업 이어받기 가능
2. **가시성**: 현재 상태가 문서로 명확히 표현됨
3. **추적성**: 의사결정 근거가 ADR로 보존됨
4. **병렬 작업 안전성**: 각 에이전트의 담당 영역이 명확히 분리됨
5. **사람도 읽을 수 있음**: 문서이므로 사람도 언제든 검토 가능

---

## Alternatives Considered (검토한 대안)

### 대안 1: 에이전트 메모리/컨텍스트 공유
- **설명**: 에이전트 간 메모리나 벡터 DB를 통해 컨텍스트 공유
- **장점**: 더 풍부한 컨텍스트 전달 가능
- **단점**: 인프라 의존성, 에이전트 교체 시 여전히 소실 위험, 불투명성
- **채택하지 않은 이유**: 외부 인프라 없이 동작해야 하며, 사람도 읽을 수 있어야 함

### 대안 2: 코드 코멘트 기반 상태 추적
- **설명**: 코드 내 TODO, FIXME 코멘트로 상태 추적
- **장점**: 코드와 상태가 가까이 있음
- **단점**: 개발 초기(코드 없는 시점)에 사용 불가, 전체 그림 파악 어려움
- **채택하지 않은 이유**: 구조화된 추적이 불가능, 재진입 프로토콜 구현 어려움

### 대안 3: 이슈 트래커(GitHub Issues) 사용
- **설명**: GitHub Issues 로 작업 상태 추적
- **장점**: 표준적인 방법
- **단점**: 외부 서비스 의존, 로컬 개발 시 접근 복잡
- **채택하지 않은 이유**: 자급자족적인 문서 패키지가 더 이식성 높음

## Consequences (결과 및 영향)

### 긍정적 영향
- 에이전트 중단/재개가 안정적
- 의사결정 근거가 보존됨
- 개발 진행 상태가 투명함
- 사람과 AI 모두 읽을 수 있음

### 부정적 영향 / 트레이드오프
- 문서 갱신 오버헤드 발생
- 에이전트가 반드시 문서 갱신 규칙을 따라야 함 (강제성 없음)
- 문서가 오래되거나 잘못될 경우 다음 에이전트에게 잘못된 컨텍스트 전달 위험

### 수용해야 할 제약
- 모든 에이전트는 `OPERATIONS_RULES.md` 를 준수해야 함
- 상태 갱신 없이 작업을 종료하는 것은 규칙 위반
